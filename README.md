# Выезды гостей санатория

Flutter/Supabase APK-приложение для организации выездов гостей санатория.

Главный принцип системы:

> Актуальным считается только график в приложении. Всё остальное — копия.

Проект не использует Google Таблицы, Google Forms, Google Apps Script и WebView как основную систему.

## Что входит

- Flutter/Dart source code.
- Supabase/PostgreSQL SQL migration.
- Seed-данные: машины, водители, пользователи и PIN-коды.
- Android APK workflow через GitHub Actions.
- `.env.example`.
- README и чек-лист проверки.
- Карточный интерфейс на русском языке.
- Роли: регистратор, корпус, водитель, главный механик, руководитель.
- Журнал действий.
- Hive-очередь для локального сохранения действий при ошибке сети.

## Роли и PIN-коды

| PIN | Роль | Пользователь |
|---|---|---|
| 1111 | Регистратор | Регистратор |
| 3001 | Корпус | Администратор корпуса 1 |
| 3002 | Корпус | Администратор корпуса 2 |
| 3003 | Корпус | Администратор корпуса 3 |
| 2101 | Водитель | Павел |
| 2102 | Водитель | Виктор |
| 2103 | Водитель | Александр |
| 5000 | Главный механик | Главный механик |
| 9000 | Руководитель | Руководитель |

PIN-коды хранятся в таблице `public.users`, их можно изменить в Supabase.

## Быстрый запуск Supabase

1. Создайте новый проект в Supabase.
2. Откройте **SQL Editor**.
3. Выполните файл:

   ```sql
   supabase/migrations/001_schema.sql
   ```

4. Затем выполните:

   ```sql
   supabase/seed.sql
   ```

5. Откройте **Project Settings → API**.
6. Скопируйте:
   - Project URL;
   - anon public key.

## Локальный запуск Flutter

```bash
flutter pub get

flutter run \
  --dart-define=SUPABASE_URL=https://your-project.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-anon-key
```

Для локальной сборки debug APK:

```bash
flutter build apk --debug \
  --dart-define=SUPABASE_URL=https://your-project.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-anon-key
```

Для release APK:

```bash
flutter build apk --release \
  --dart-define=SUPABASE_URL=https://your-project.supabase.co \
  --dart-define=SUPABASE_ANON_KEY=your-anon-key
```

Файл появится здесь:

```text
build/app/outputs/flutter-apk/app-release.apk
```

## Настройка GitHub Actions

1. Создайте репозиторий на GitHub.
2. Загрузите туда весь проект.
3. Откройте **Settings → Secrets and variables → Actions**.
4. Добавьте два секрета:

   ```text
   SUPABASE_URL
   SUPABASE_ANON_KEY
   ```

5. Откройте вкладку **Actions**.
6. Запустите workflow **Build Flutter APK** вручную или сделайте push в `main`.
7. После завершения откройте запуск workflow.
8. Внизу страницы скачайте artifact:

   ```text
   sanatorium-trips-apk
   ```

Внутри будет APK:

```text
app-release.apk
```

## Логика работы

### Корпус

1. Входит по PIN `3001`, `3002` или `3003`.
2. Подаёт заявку на выезд.
3. Может отправить изменение или отмену.
4. Видит только заявки своего корпуса.

### Регистратор

1. Входит по PIN `1111`.
2. Видит все заявки.
3. Создаёт рейс из заявки.
4. Назначает машину, водителя и время.
5. Видит главный график, проблемы, срочные изменения, журнал и архив.
6. Закрывает рейсы.

### Водитель

1. Входит по PIN `2101`, `2102` или `2103`.
2. Видит только свои рейсы на сегодня или завтра.
3. Работает с четырьмя крупными кнопками:
   - `ПРИНЯЛ`;
   - `МАШИНА ПОДАНА`;
   - `ГОСТИ УЕХАЛИ`;
   - `ПРОБЛЕМА`.

### Главный механик

1. Входит по PIN `5000`.
2. Видит транспорт на сегодня и завтра.
3. Отмечает:
   - `ГОТОВА`;
   - `НЕ ГОТОВА`;
   - `НУЖНА ЗАМЕНА`;
   - `ПРОБЛЕМА`.

Если машина не готова, связанные рейсы получают статус `Проблема`.

### Руководитель

1. Входит по PIN `9000`.
2. Видит график, панель контроля, журнал и архив.
3. Не редактирует данные.

## ID рейсов

ID создаётся в PostgreSQL автоматически триггером `trg_set_trip_code`.

Формат:

```text
ДД-ММ-НН
```

Пример:

```text
21-05-01
21-05-02
21-05-03
```

## Offline-режим

Первая версия делает минимум, необходимый для безопасной работы:

- если нет связи, приложение показывает сообщение `Нет связи с сервером`;
- введённая заявка или изменение сохраняется локально в Hive;
- приложение не падает при сетевой ошибке;
- количество ожидающих локальных действий видно в главном меню.

Для промышленного режима рекомендуется добавить автоматическую фоновую синхронизацию очереди с конфликт-резолвингом.

## Важное по безопасности

Эта версия использует простой PIN-вход и Supabase anon key. Это удобно для внутреннего служебного APK, но PIN не является полноценной криптографической авторизацией.

Для следующего этапа рекомендуется:

- перевести пользователей на Supabase Auth;
- сделать серверные RPC/Edge Functions для действий по ролям;
- ужесточить RLS-политики;
- хранить PIN в виде хеша, а не открытого текста;
- подписать release APK своим keystore.

## Главный рабочий сценарий

```text
Корпус → заявка в приложении → регистратор создаёт рейс → водитель видит свой рейс → механик подтверждает машину → водитель выполняет рейс → всё сохраняется в базе и журнале.
```
